## 슬라이드 2: 프로젝트 소개 — 62댕냥이란?

**시각 자료:** 플랫폼 메인 화면 스크린샷 / 기술 스택 로고 배치

> 먼저 프로젝트를 간단히 소개드리겠습니다.
>
> 62댕냥은 **유기동물 입양·보호 플랫폼**입니다. 쉽게 말하면 **"동물을 위한 직방"**이라고 생각하시면 됩니다. 직방이 집을 찾아주듯이, 저희 플랫폼은 유기동물의 새 가족을 찾아주는 서비스예요.
>
> 공공 데이터 포털에서 유기동물 정보를 자동으로 가져와서 사용자에게 보여주고, 사용자가 마음에 드는 동물을 만나면 입양이나 임시보호를 신청할 수 있습니다.
>
> 기술 스택을 보면, 백엔드는 **Java 17 + Spring Boot 3** — 자바 진영에서 가장 많이 쓰이는 조합이고요, 프론트엔드는 **React + TypeScript** — 요즘 가장 인기 있는 프론트엔드 기술입니다. 데이터베이스는 **PostgreSQL**을 사용합니다.

## 슬라이드 3: 입양 vs 임시보호 — 무엇이 다를까?

**시각 자료:** 입양(🟢)과 임시보호(🔵)를 나란히 비교하는 인포그래픽

> 코드를 보기 전에, 먼저 **"입양"**과 **"임시보호"**가 뭐가 다른지 아셔야 합니다.
>
> **입양**은 말 그대로 동물의 영구적인 새 가족이 되는 거예요. 법적 소유권이 넘어오고, 평생 책임지는 겁니다.
>
> **임시보호**는 좀 다릅니다. 보호소가 너무 꽉 찼을 때, 일정 기간 동안 동물을 대신 돌봐주는 거예요. 기간이 끝나면 보호소로 돌아가거나, 그 사이에 입양으로 전환되기도 합니다.
>
> 여기서 중요한 **설계 포인트**가 있습니다. 입양과 임시보호는 절차가 거의 비슷하거든요 — 둘 다 "신청 → 심사 → 승인/거절"의 같은 흐름이에요. 그래서 우리 코드에서는 **`AdoptionType`이라는 하나의 Enum**으로 구분합니다. 테이블도 하나로 통합했어요. 이렇게 하면 코드 중복이 확 줄어듭니다.

## 슬라이드 4: 전체 아키텍처 — 3층 건물

**시각 자료:** 3-Layer 아키텍처 다이어그램 (건물 층수 비유 포함)

> 이제 전체 구조를 보겠습니다. **3층짜리 건물**을 상상해보세요.
>
> **1층은 안내 데스크**, React 프론트엔드입니다. 사용자가 처음 만나는 곳이에요. 예쁜 화면을 보여주고, 사용자의 요청을 받아서 2층으로 올려보냅니다.
>
> **2층은 담당 직원**, Spring Boot REST API입니다. 실제 업무를 처리하는 곳이에요. "이 사람이 입양 자격이 되는지 확인하고, 서류를 작성해서 보관함에 넣어라" 같은 일을 합니다.
>
> **3층은 서류 보관함**, PostgreSQL 데이터베이스입니다. 모든 신청 정보, 사용자 정보, 동물 정보가 여기에 저장됩니다.
>
> 구청 민원실을 생각하시면 딱입니다. 1층에서 접수하고 → 2층에서 처리하고 → 3층에 보관하는 거죠.

## 슬라이드 5: 데이터 모델 — Adoption Entity

**시각 자료:** ER 다이어그램 (Adoption ↔ User, Animal 관계) 또는 서류 양식 비유 이미지

> 이제 데이터가 어떻게 생겼는지 볼까요? **Adoption Entity**가 바로 입양 신청서의 양식입니다.
>
> DB에 `adoptions`라는 테이블이 있고, JPA Entity가 이 테이블과 매핑됩니다.
>
> 중요한 필드만 보면요:
> - **id** — 접수번호예요. 자동으로 1, 2, 3 증가합니다.
> - **user** — 누가 신청했는지, 신청자입니다.
> - **animal** — 어떤 동물인지, 대상 동물입니다.
> - **type** — 입양인지 임시보호인지 구분합니다.
> - **status** — 처리 상태예요. 이건 다음 슬라이드에서 자세히 볼게요.
> - **reason** — "왜 이 동물을 입양하고 싶은가요?" 신청 사유입니다.
> - **familyAgreement** — 가족 동의 여부 체크란이에요.
>
> 기억하세요: **Entity는 종이 서류 양식**, **Repository는 서류 보관함**, **Service는 담당 직원**입니다.

## 슬라이드 6: 상태 머신 — 신청서의 생애주기

**시각 자료:** 상태 전이 다이어그램 (PENDING → 3가지 최종 상태)

> 이 슬라이드가 입양 기능에서 **가장 중요한 개념**입니다.
>
> 모든 신청은 **PENDING(대기)** 상태에서 시작합니다. 여기서 세 가지 갈림길이 있어요.
>
> 1. 관리자가 "좋아, 승인!" → **APPROVED**
> 2. 관리자가 "죄송하지만 거절" → **REJECTED**
> 3. 사용자가 "저 취소할게요" → **CANCELLED**
>
> **핵심 규칙**이 있습니다. APPROVED, REJECTED, CANCELLED는 모두 **최종 상태**입니다. 한번 도장이 찍히면 되돌릴 수 없어요.
>
> 왜 이렇게 할까요? 실생활에서 생각해보세요. 이미 서류에 "승인" 도장을 찍었는데, 그걸 취소하고 다시 대기로 돌리면 혼란이 생기겠죠? 택배로 비유하면, 이미 배송 완료된 물건을 "배송 중"으로 되돌릴 수 없는 것과 같습니다.

## 슬라이드 7: API 설계 — 6개 엔드포인트

**시각 자료:** REST 엔드포인트 목록 표

> API는 프론트엔드와 백엔드가 통신하는 **약속**입니다. "이런 주소로, 이런 데이터를 보내면, 이런 결과를 줄게"라는 계약이에요.
>
> 입양 기능에는 **6개의 엔드포인트**가 있습니다.
>
> 먼저 사용자용 3개:
> - **POST /api/adoptions** — 새로운 입양/임보를 신청합니다.
> - **GET /api/adoptions/my** — 내가 신청한 목록을 불러옵니다.
> - **PUT /{id}/cancel** — 대기 중인 내 신청을 취소합니다.
>
> 그리고 관리자용 3개:
> - **GET /shelter/pending** — 보호소에 들어온 대기 신청 목록을 봅니다.
> - **PUT /{id}/approve** — 승인합니다.
> - **PUT /{id}/reject** — 거절합니다. 이때 거절 사유도 함께 보냅니다.
>
> 모든 API는 **JWT 토큰**이 필요합니다. 즉, 로그인해야만 사용할 수 있어요.

## 슬라이드 8: 신청 프로세스 — 사용자 관점

**시각 자료:** 사용자 여정 흐름도 (5단계 순서도)

> 사용자 입장에서 입양 신청이 어떻게 진행되는지 보겠습니다.
>
> 쇼핑에 비유하면 아주 직관적입니다.
>
> 1. **동물 탐색** — 온라인 쇼핑몰에서 상품 둘러보는 것처럼, 동물 목록을 필터링해서 봅니다.
> 2. **상세 보기** — 마음에 드는 동물을 클릭하면 상세 정보가 나옵니다.
> 3. **절차 안내** — "입양 절차 안내" 버튼을 누르면 가이드 페이지로 이동합니다. 6단계 절차가 나와요.
> 4. **신청/연락** — 보호소 전화번호를 확인하고 연락하거나, 온라인으로 신청합니다.
> 5. **확인 이메일 수신** — 신청이 접수되면 이메일로 "신청이 접수되었습니다" 알림을 받습니다.
>
> 이후에는 관리자가 검토할 때까지 기다리면 됩니다.

## 슬라이드 9: 신청 프로세스 — 코드 관점

**시각 자료:** 코드 스니펫 하이라이트 (apply 메서드 핵심 4단계)

> 이제 코드가 실제로 어떻게 동작하는지 볼까요? `AdoptionService.apply()` 메서드입니다.
>
> 크게 **4단계**입니다.
>
> **1단계** — 사용자와 동물을 DB에서 찾습니다. "이 사용자가 정말 존재하는지, 이 동물이 정말 있는지" 확인하는 거예요. 없으면 404 에러가 발생합니다.
>
> **2단계** — 신청서(Entity)를 작성합니다. 빌더 패턴으로 깔끔하게 만들어요. 이때 **상태는 항상 PENDING**으로 시작합니다.
>
> **3단계** — DB에 저장합니다. 서류를 보관함에 넣는 거죠. `adoptionRepository.save()`입니다.
>
> **4단계** — 부수 효과가 실행됩니다. 신청자에게 접수 확인 이메일을 보내고, 보호소 관리자에게 "새로운 신청이 들어왔어요" 알림을 보냅니다.
>
> 이 4단계가 `@Transactional` 안에서 하나로 묶여서 실행됩니다.

## 슬라이드 10: 관리자 승인/거절 프로세스

**시각 자료:** approve/reject 코드 스니펫 나란히 비교

> 관리자가 처리하는 부분을 보겠습니다. 승인과 거절을 나란히 비교하면 구조가 거의 같아요.
>
> **공통 동작** 세 가지:
> 1. 먼저 PENDING 상태인지 확인합니다. 이미 처리된 건은 다시 처리할 수 없어요.
> 2. `processedAt`에 현재 시각을 기록합니다. "몇 시에 처리했나" 기록이죠.
> 3. 이메일을 발송합니다.
>
> **차이점**은 하나입니다. `reject()`는 **거절 사유(`rejectReason`)를 반드시 저장**해야 합니다. 이 사유는 DB에 영구 보관되고, 신청자에게 보내는 이메일에도 포함됩니다.
>
> 나중에 "왜 이 신청을 거절했지?" 추적할 수 있도록 기록을 남기는 거예요. 실무에서 매우 중요한 패턴입니다.

## 슬라이드 11: 알림 시스템 연동

**시각 자료:** 알림 발송 시퀀스 다이어그램 (4가지 시점)

> 알림 시스템을 보겠습니다. 총 **4가지 시점**에서 알림이 발송됩니다.
>
> **신청 접수 시:**
> - 신청자에게 → 이메일로 "접수 확인" 안내
> - 보호소 관리자에게 → 이메일 + 인앱 알림으로 "새 신청이 들어왔습니다" 알림
>
> **승인 시:** 신청자에게 → 이메일로 "축하합니다, 승인되었습니다"
> **거절 시:** 신청자에게 → 이메일로 "거절되었습니다" + 거절 사유
>
> 이메일 발송은 **Resend API**를 사용하고, 인앱 알림은 DB에 저장하는 방식입니다.
>
> 참고로 관리자에게만 이메일과 인앱 알림을 둘 다 보내는 이유는, 새로운 신청을 빠르게 인지하도록 하기 위해서예요.

## 슬라이드 12: 프론트엔드 — API 모듈

**시각 자료:** 코드 스니펫 하이라이트 (adoption.ts)

> 프론트엔드에서 백엔드 API를 호출하는 코드를 보겠습니다. `adoption.ts` 파일인데요.
>
> 이 파일을 **전화기**에 비유해볼게요. 프론트엔드가 백엔드에 "전화"를 걸 때 사용하는 번호부입니다.
>
> 각 함수가 백엔드 API와 1:1로 대응합니다.
> - `apply()` → POST로 신청
> - `getMyList()` → GET으로 내 목록 조회 (페이징 지원)
> - `cancel()` → PATCH로 취소
> - `approve()` → PUT으로 승인
> - `reject()` → PUT으로 거절 (거절 사유 포함)
>
> **Axios 인스턴스**를 사용하기 때문에 JWT 토큰과 baseURL이 자동으로 적용됩니다. 매번 토큰을 수동으로 붙일 필요가 없어요.

## 슬라이드 13: 프론트엔드 — 입양 가이드 페이지

**시각 자료:** GuideAdoptionPage 화면 목업 / 6단계 흐름도

> 입양 가이드 페이지, `GuideAdoptionPage`입니다.
>
> 사용자가 "입양하고 싶다!" 할 때 안내해주는 페이지예요. **6단계 절차**를 보여줍니다.
>
> 1단계: 동물사랑배움터에서 필수교육을 이수합니다.
> 2단계: 보호 중인 동물을 확인합니다.
> 3단계: 보호소에 상담을 예약합니다.
> 4단계: 실제로 동물을 만나봅니다.
> 5단계: 입양 계약서를 작성하고 확정합니다.
> 6단계: 입양 후 사후 관리를 받습니다.
>
> 여기서 재밌는 기능이 있는데요, URL에 `animalId` 파라미터가 있으면 해당 동물 정보를 자동으로 로드해서 보여줍니다. 그리고 **GuideCallBlock** 컴포넌트가 보호소 전화번호와 "이렇게 전화하세요" 예시 문구까지 제공합니다.

## 슬라이드 14: 프론트엔드 — 임시보호 가이드 페이지

**시각 자료:** GuideFosterPage 화면 목업 / 4단계 흐름도

> 임시보호 가이드 페이지는 입양과 비슷하지만, 절차가 더 간소합니다. **4단계**예요.
>
> 1단계: 신청서를 작성합니다.
> 2단계: 보호소에서 가정환경을 점검합니다.
> 3단계: 계약서를 작성합니다.
> 4단계: 동물을 인수합니다.
>
> 디자인 구조는 입양 가이드와 동일하고, 색상만 **파란색 테마**로 다릅니다. 그리고 같은 `GuideCallBlock` 컴포넌트를 재사용해요. 코드를 새로 짜지 않고 기존 것을 다시 쓰는 거죠.

## 슬라이드 15: GuideCallBlock — 공용 전화연결 컴포넌트

**시각 자료:** GuideCallBlock 컴포넌트 화면 캡처 (녹색/파란색 두 버전)

> 이 컴포넌트를 따로 설명드리는 이유가 있습니다. **재사용 가능한 컴포넌트 설계**의 좋은 예시이기 때문이에요.
>
> GuideCallBlock은 입양 가이드에도 쓰이고, 임시보호 가이드에도 쓰입니다. 같은 컴포넌트인데 **녹색 케이스(입양)**와 **파란색 케이스(임시보호)**를 바꿔 끼우는 것처럼요.
>
> 이 컴포넌트가 보여주는 정보:
> - 동물의 유기번호
> - 보호소명
> - 담당자 전화번호
>
> 그리고 "예시 문구 복사" 버튼이 있어서, 전화할 때 뭐라고 말하면 좋을지 클립보드에 복사해줍니다. 사용자 입장에서 참 친절한 기능이죠.

## 슬라이드 16: 검증 & 에러 핸들링

**시각 자료:** 코드 스니펫 하이라이트 (cancel 메서드의 2중 검증)

> 보안과 검증 이야기를 해볼게요.
>
> `cancel()` 메서드를 보면 **2중 검증**을 합니다. 이건 **방어적 프로그래밍**이라고 해요.
>
> 식당 예약 취소에 비유하면 이해가 쉽습니다.
>
> **첫 번째 검증: "본인 확인"** — 다른 사람 예약을 내가 취소할 수 없죠? 코드에서는 `userId`를 비교해서 본인이 아니면 **403 Forbidden**을 던집니다.
>
> **두 번째 검증: "취소 가능한 상태인가?"** — 이미 당일 방문해서 식사를 했다면 예약 취소가 의미가 없죠? 코드에서는 PENDING 상태가 아니면, 즉 이미 승인/거절된 건이면 **400 Bad Request**를 던집니다.
>
> 이렇게 2중으로 검증하면 잘못된 요청이 서버까지 도달하더라도 안전하게 막을 수 있습니다.

## 슬라이드 17: 입양 vs 임시보호 — 왜 하나의 테이블?

**시각 자료:** 분리 설계 vs 통합 설계 비교 다이어그램

> 설계 결정에 대해 이야기하겠습니다. 왜 입양과 임시보호를 **하나의 테이블로 통합**했을까요?
>
> 만약 **분리**했다면? `adoption_table`과 `fostering_table` 두 개가 필요하고, 컬럼은 거의 같고, Controller도 2개, Service도 2개 만들어야 합니다. 뭔가 수정하려면 양쪽 다 고쳐야 해요.
>
> 하지만 **통합**하면? 테이블 1개, Enum 하나(`ADOPTION` vs `FOSTERING`)로 구분하면 끝! Controller도 Service도 하나씩이면 충분합니다.
>
> 병원에 비유하면요, "외래 접수증"과 "입원 접수증"을 따로 만들 필요 없이, 하나의 접수증 양식에 "외래/입원" 체크란만 추가하면 되는 거예요.
>
> 이것이 실무에서 **코드 중복을 줄이는** 핵심 설계 원칙입니다.

## 슬라이드 18: 전체 시퀀스 다이어그램

**시각 자료:** 신청 → 이메일/알림 → 관리자 승인까지 전체 시퀀스 다이어그램

> 지금까지 배운 것을 하나의 흐름으로 연결해보겠습니다.
>
> **1단계: 사용자가 신청합니다.**
> 브라우저에서 `adoption.ts`를 통해 HTTP 요청이 날아가고, Controller → Service → Repository 순서로 처리됩니다. DB에 저장되면서 동시에 신청자에게 이메일이, 관리자에게 알림이 갑니다.
>
> **2단계: 관리자가 처리합니다.**
> 관리자 대시보드에서 대기 목록을 보고, 승인 또는 거절 버튼을 누릅니다. 백엔드에서 상태가 변경되고, 결과 이메일이 신청자에게 발송됩니다.
>
> 이 두 단계가 입양 기능의 전부입니다. 간단하죠?

## 슬라이드 19: 정리 — 오늘 배운 핵심 5가지

**시각 자료:** 핵심 5가지 요약 불렛 포인트 + 비유 키워드

> 마지막으로 핵심을 정리하겠습니다.
>
> **하나**, 3-Layer 아키텍처. React가 안내 데스크, Spring Boot가 담당 직원, DB가 서류 보관함입니다.
>
> **둘**, 통합 설계. 입양과 임시보호를 `AdoptionType` Enum 하나로 구분해서 하나의 테이블로 관리합니다. 중복을 줄이는 실무 설계죠.
>
> **셋**, 상태 머신. PENDING에서 시작해서 APPROVED, REJECTED, CANCELLED 세 가지 최종 상태로 갈 수 있고, 한번 결정되면 되돌릴 수 없습니다.
>
> **넷**, 알림 자동화. 신청/승인/거절 시 이메일과 인앱 알림이 자동으로 발송됩니다.
>
> **다섯**, 방어적 프로그래밍. 2중 검증(권한 + 상태)으로 잘못된 요청을 안전하게 차단합니다.
>
> 이 다섯 가지를 기억하시면 입양 기능 전체를 이해한 겁니다!
