🏠 62댕냥이란?

  • 유기동물 입양·보호 플랫폼 — 보호소의 동물 정보를 한곳에 모아 입양 연결
  • 공공 API 데이터(유기동물 정보)를 자동 수집 → 사용자에게 제공
  • 사용자가 입양/임시보호를 온라인으로 신청하고, 관리자가 승인/거절

백엔드

  • Java 17 + Spring Boot 3
  • JPA/Hibernate + PostgreSQL

프론트엔드

  • React + TypeScript
  • Vite + TailwindCSS
💡 비유: 62댕냥 = "동물을 위한 직방" — 집 대신 동물의 새 가족을 찾아주는 플랫폼

🐕 입양 vs 임시보호 — 무엇이 다를까?

🏡 입양 (Adoption)

  • 동물의 영구적인 새 가족이 되는 것
  • 법적 소유권 이전
  • 평생 책임

🏘️ 임시보호 (Fostering)

  • 일정 기간 동물을 돌봐주는 것
  • 보호소 과밀 해소 목적
  • 기간 종료 후 반환 또는 입양 전환
설계 포인트: 코드에서는 AdoptionType Enum 하나로 통합 관리 → 테이블도 하나! (코드 중복 ↓)

🏗️ 전체 아키텍처 — 3층 건물

🖥️ React 프론트엔드 (1층: 안내 데스크)
↕ HTTP 요청/응답
⚙️ Spring Boot REST API (2층: 담당 직원)
↕ JPA/SQL
🗄️ PostgreSQL DB (3층: 서류 보관함)
💡 비유: 구청 민원실과 같아요. 1층 안내 데스크(프론트)에서 서류 접수 → 2층 담당 직원(서비스)이 처리 → 3층 서류 보관함(DB)에 저장

🗄️ 데이터 모델 — Adoption Entity

DB의 adoptions 테이블 = "입양 신청서 양식"

필드 타입 비유
id Long (자동 증가) 접수번호
user User (다대일) 신청자 이름
animal Animal (다대일) 어떤 동물인지
type Enum: ADOPTION / FOSTERING 입양 or 임시보호
status Enum: PENDING / APPROVED / … 처리 상태 도장
reason String "왜 입양하고 싶은가요?"
familyAgreement Boolean 가족 동의 여부 체크란
💡 Entity = 종이 서류 양식, Repository = 서류 보관함, Service = 담당 직원

🔄 상태 머신 — 신청서의 생애주기

PENDING (대기)
↙          ↓          ↘
APPROVED
(승인)     REJECTED
(거절)     CANCELLED
(취소)
핵심 규칙: APPROVED / REJECTED / CANCELLED는 모두 최종 상태 (되돌릴 수 없음)
💡 이미 도장 찍힌 서류는 수정할 수 없는 것처럼, 한번 처리된 신청은 상태를 되돌릴 수 없습니다.

📡 API 설계 — 6개 엔드포인트

HTTP 메서드 경로 설명 누가?
POST /api/adoptions 입양/임보 신청 로그인 사용자
GET /api/adoptions/my 내 신청 목록 로그인 사용자
PUT /api/adoptions/{id}/cancel 신청 취소 본인만
GET /api/adoptions/shelter/pending 보호소 대기 목록 보호소 관리자
PUT /api/adoptions/{id}/approve 승인 관리자
PUT /api/adoptions/{id}/reject 거절 관리자
모든 API는 /api/adoptions 하위 · JWT Bearer 토큰 필수 (로그인 필요!)

📝 신청 프로세스 — 사용자 관점

  1. 동물 목록에서 마음에 드는 동물 클릭
  2. 상세 페이지에서 "입양 절차 안내" 버튼 클릭
  3. 입양 가이드 6단계 확인 (교육 이수 → 상담 → 만남 → 계약)
  4. 보호소 전화번호 확인 후 직접 연락 또는 온라인 신청
  5. 신청 접수 → 이메일로 접수 확인 알림 수신
동물 검색 → 상세 페이지 → 절차 안내 → 신청 접수 → 대기
💡 온라인 쇼핑처럼 — 상품 탐색 → 상세 보기 → 주문(신청) → 주문 확인 이메일 수신

⚙️ 신청 프로세스 — 코드 관점

AdoptionService.apply() — 가장 핵심 메서드 

@Transactional
public AdoptionResponse apply(Long userId, AdoptionRequest request) {
    // 1️⃣ 사용자 & 동물 조회 (없으면 404 에러)
    User user = userRepository.findById(userId).orElseThrow(...);
    Animal animal = animalRepository.findById(request.getAnimalId()).orElseThrow(...);

    // 2️⃣ 신청서(Entity) 작성 — 상태는 항상 PENDING으로 시작
    Adoption adoption = Adoption.builder()
        .user(user).animal(animal)
        .type(request.getType())    // ADOPTION 또는 FOSTERING
        .status(AdoptionStatus.PENDING)
        .build();

    // 3️⃣ DB에 저장 (서류 보관함에 넣기)
    adoption = adoptionRepository.save(adoption);

    // 4️⃣ 이메일 발송 + 관리자 알림 (부수 효과)
    emailService.sendApplicationReceivedEmail(...);
    notifyAndEmailAdmin(adoption, ...);

    return toResponse(adoption);
}

👨‍💼 관리자 승인/거절 프로세스

✅ approve() — 승인

// PENDING 상태 확인
adoption.setStatus(APPROVED);
adoption.setProcessedAt(now());
// 📧 승인 이메일 발송
emailService.sendApprovalEmail(...);

❌ reject() — 거절

// PENDING 상태 확인
adoption.setStatus(REJECTED);
adoption.setRejectReason(reason);
adoption.setProcessedAt(now());
// 📧 거절 이메일 (사유 포함)
emailService.sendRejectionEmail(...);
공통점: 둘 다 PENDING 상태에서만 가능 · processedAt 기록 · 이메일 발송
차이점: 거절 시 rejectReason을 반드시 저장

🔔 알림 시스템 연동

발송되는 알림 4가지

시점 받는 사람 방식 내용
신청 접수 신청자 📧 이메일 "접수 확인" 안내
신청 접수 보호소 관리자 📧 이메일 + 🔔 인앱 "새 신청이 들어왔습니다"
승인 신청자 📧 이메일 "승인되었습니다" 축하
거절 신청자 📧 이메일 "거절되었습니다" + 사유
EmailService → Resend API로 이메일 발송  |  NotificationService → DB에 인앱 알림 저장 (타입: ADOPTION_APPLICATION)

🔌 프론트엔드 — API 모듈 (adoption.ts)

export const adoptionApi = {
  // 📝 입양/임보 신청
  apply: (data: AdoptionRequest) =>
    axiosInstance.post('/adoptions', data),

  // 📋 내 신청 목록 (페이징)
  getMyList: (page = 0, size = 10) =>
    axiosInstance.get('/adoptions/my', { params: { page, size } }),

  // ❌ 신청 취소
  cancel: (id: number) =>
    axiosInstance.patch(`/adoptions/${id}/cancel`),

  // ✅ 승인 (관리자)
  approve: (id: number) =>
    axiosInstance.put(`/adoptions/${id}/approve`),

  // 🚫 거절 (관리자)
  reject: (id: number, rejectReason?: string) =>
    axiosInstance.put(`/adoptions/${id}/reject`, { rejectReason }),
};
💡 비유: adoption.ts = "전화기" — 프론트엔드가 백엔드에 전화 걸 때 쓰는 번호부

📖 입양 가이드 페이지 — 6단계 안내

GuideAdoptionPage.tsx

1️⃣ 필수교육 이수 → 2️⃣ 대기동물 확인 → 3️⃣ 상담 예약
4️⃣ 동물 만남 → 5️⃣ 계약·확정 → 6️⃣ 사후 관리
  • animalId 쿼리 파라미터로 특정 동물 정보 자동 로드
  • GuideCallBlock 컴포넌트로 보호소 전화번호 표시 + 예시 문구 복사 기능
  • 입양 = 🟢 녹색, 임시보호 = 🔵 파란색 테마

📖 임시보호 가이드 페이지 — 4단계 안내

GuideFosterPage.tsx

1️⃣ 신청서 작성 → 2️⃣ 가정환경 점검 → 3️⃣ 계약서 작성 → 4️⃣ 동물 인수
  • 구조·디자인은 입양 가이드와 동일 (파란색 테마)
  • GuideCallBlock 컴포넌트를 재사용 — 코드 중복 최소화
  • 입양 가이드(6단계)보다 간소한 절차

📞 GuideCallBlock — 공용 전화연결 컴포넌트

  • 입양 가이드 + 임보 가이드 페이지 모두에서 재사용
  • 동물의 유기번호 / 보호소명 / 담당자 전화번호 표시
  • "예시 문구 복사" 기능 — 클립보드에 전화할 때 쓸 멘트 복사
  • 입양 = 🟢 green 스타일 / 임보 = 🔵 blue 스타일로 분기
재사용 컴포넌트 설계: 하나의 컴포넌트를 만들어 여러 페이지에서 재사용 → 유지보수 용이. 마치 같은 전화기를 녹색 케이스(입양)와 파란색 케이스(임보)로 바꿔 쓰는 것!

🛡️ 검증 & 에러 핸들링

입력 검증 (@Valid)

  • animalId — @NotNull 필수
  • type — @NotNull 필수
  • familyAgreement — @NotNull 필수

비즈니스 검증

  • 본인 신청만 취소 가능 → 403
  • PENDING 상태에서만 취소 → 400
  • 존재하지 않는 신청 → 404
// cancel() 메서드의 2중 검증
if (!adoption.getUser().getId().equals(userId))
    throw new CustomException("본인의 신청만 취소", FORBIDDEN);  // 403

if (adoption.getStatus() != PENDING)
    throw new CustomException("대기 중인 신청만 취소", BAD_REQUEST);  // 400
💡 식당 예약 취소와 같아요 — "본인 예약만 취소", "아직 방문 전인 예약만 취소 가능"

🔗 입양 vs 임시보호 — 왜 하나의 테이블?

❌ 분리 설계 (비추)

  • adoption_table + fostering_table
  • 거의 같은 컬럼 중복
  • Controller도 2개, Service도 2개
  • 수정 시 양쪽 다 고쳐야 함

✅ 통합 설계 (채택)

  • adoptions 테이블 1개
  • AdoptionType Enum으로 구분
  • Controller 1개, Service 1개
  • 수정 시 한 곳만 고치면 됨!
💡 비유: 병원에서 "외래 접수증"과 "입원 접수증"을 따로 만들 필요 없이,
하나의 접수증 양식에 "외래/입원"을 체크하는 칸만 추가하면 되는 것과 같습니다.

🔄 전체 시퀀스 — 신청부터 승인까지

1. 사용자 신청
Browser → adoption.ts → Controller → Service → DB 저장
　　　　　　　　　　　　↳ 📧 신청자 이메일 + 🔔 관리자 알림

2. 관리자 처리
Admin 대시보드 → approve() / reject() → 상태 변경
　　　　　　　　　　　　↳ 📧 승인/거절 이메일 발송

📌 정리 — 오늘 배운 핵심 5가지

  1. 3-Layer 아키텍처: React → Spring Boot → PostgreSQL
  2. 통합 설계: 입양 + 임시보호를 AdoptionType Enum으로 하나의 테이블에 관리
  3. 상태 머신: PENDING → APPROVED / REJECTED / CANCELLED (최종 상태 = 되돌릴 수 없음)
  4. 알림 자동화: 신청/승인/거절 시 이메일 + 인앱 알림 자동 발송
  5. 방어적 프로그래밍: 2중 검증 (권한 + 상태) → 잘못된 요청 차단
기억할 비유: Entity = 서류 양식  |  Repository = 서류 보관함  |  Service = 담당 직원  |  Controller = 접수 창구
