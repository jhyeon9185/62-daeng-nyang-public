## [Slide 3] '외부 API'란?

### 제목
**외부 API 연동이란?**

### 핵심 내용
- **API (Application Programming Interface)**: 프로그램끼리 대화하는 방법
- **비유**: 우리 도서관(내 서버)에 없는 책을 국회도서관(공공데이터포털)에서 빌려와 보여주는 것
- **필수 요소**: 요청(Request), 응답(Response), 키(Key - 출입증)

### 시각 자료 제안
- 도서관 사서(내 서버)가 다른 거대 도서관(외부 API)에 전화를 걸어 책 정보를 물어보는 일러스트

### 대본 (스크립트)
여러분, '외부 API 연동'이라는 말이 어렵게 느껴지시나요?
이렇게 생각해보세요. 여러분이 작은 동네 도서관을 운영한다고 칩시다. 그런데 우리 도서관에 없는 희귀한 책을 손님이 찾아요. 그럼 어떻게 하죠? 국회도서관 같은 큰 곳에 전화해서 "그 책 있나요?" 물어보고, 정보를 받아서 손님한테 알려주겠죠?
여기서 '전화 거는 것'이 API 요청(Request), '정보를 받는 것'이 API 응답(Response)입니다. 그리고 아무나 전화하면 안 되니까 '도서관 회원증' 같은 걸 보여줘야 하는데, 그게 바로 API Key입니다.
우리는 공공데이터포털이라는 아주 큰 도서관에서 '유기동물 정보'라는 책 목록을 매일매일 빌려올 겁니다.

## [Slide 5] 공공 API 호출 — PublicApiService

### 제목
**데이터 가져오기: PublicApiService**

### 핵심 내용
- **URL**: `https://apis.data.go.kr/...`
- **Method**: HTTP GET
- **Parameter**: `serviceKey`(인증), `numOfRows`(개수), `uprCd`(지역) 등
- **Parsing**: JSON 형식의 응답을 Java 객체로 변환

### 시각 자료 제안
- 코드 스니펫: `UriComponentsBuilder`를 사용하여 URL을 만드는 부분과 `restTemplate.exchange()` 호출 부분 (핵심 3~4줄)

### 대본 (스크립트)
자, 이제 코드를 살짝 볼까요? `PublicApiService`는 '외교관' 같은 역할을 합니다.
외부 세계(공공데이터포털)와 소통하죠. 자바 코드에서 `UriComponentsBuilder`를 사용해 주소를 만들고, 서비스 키(ServiceKey)라는 출입증을 붙여서 HTTP GET 요청을 보냅니다.
그러면 공공데이터포털이 JSON이라는 형식으로 응답을 주는데요, 이 복잡한 문자를 자바가 이해할 수 있는 객체로 알맞게 바꾸는 작업도 여기서 수행합니다.

## [Slide 6] API 응답 파싱의 함정

### 제목
**개발자를 울리는 API 응답의 함정**

### 핵심 내용
- **문제**: 데이터가 1개일 때는 `Object`, 2개 이상일 때는 `Array`로 옴
- **해결**: `Jackson` 라이브러리의 유연함 또는 커스텀 Deserializer 필요
- **구조**: `response` > `body` > `items` > `item` (깊은 중첩 구조)

### 시각 자료 제안
- JSON 예시 비교: 왼쪽은 `{ item: { ... } }` (객체), 오른쪽은 `{ item: [ ... ] }` (배열)
- 곤란해하는 개발자 이모지 🤯

### 대본 (스크립트)
그런데 여기서 잠깐! 개발하다 보면 멘붕(?)에 빠지는 순간이 옵니다.
공공 데이터가 참 친절하지 않을 때가 있어요. 검색 결과가 여러 개면 `[목록]`(배열)으로 주는데, 딱 1개만 있으면 목록이 아니라 그냥 `{객체}` 하나만 덜렁 줍니다.
우리 코드는 "무조건 목록(List)이겠지?" 하고 기다리는데 갑자기 객체가 오면 에러가 빵! 터지죠.
그래서 우리는 `PublicApiResponse`라는 똑똑한 포장지를 만들어서, 1개가 오든 여러 개가 오든 알아서 척척 처리하도록 코드를 짰습니다. 이런 게 바로 실무 노하우죠!

### 제목

### 핵심 내용

- **매핑**: JSON 필드(`desertionNo`) ↔ Java 필드(`desertionNo`)
- **어노테이션**: `@JsonProperty`, `@JsonAlias` 등으로 이름 불일치 해결

### 시각 자료 제안
- 표(Table): JSON 필드명(desertionNo, happenDt)과 Java 필드명 매핑 보여주기
- 코드 스니펫: `AnimalItem` 클래스의 필드 선언부

### 대본 (스크립트)
공공 데이터가 보내준 JSON은 그냥 문자열 덩어리일 뿐입니다. 이걸 우리가 쓰기 편한 자바 객체로 바꿔야 하는데, 이때 등장하는 것이 `AnimalItem` 클래스입니다.
"유기번호는 `desertionNo`에 담고, 접수일은 `happenDt`에 담아라"라고 딱 정해주는 거죠.
마치 해외 직구로 물건을 샀을 때, 송장에 적힌 영어 주소를 보고 한국 택배 기사님이 알아보실 수 있게 한글 주소 라벨을 붙이는 과정과 비슷합니다.

## [Slide 8] 동기화의 핵심 — AnimalSyncService (1)

### 제목
**동기화의 마법: Upsert 패턴**

### 핵심 내용
- **Upsert** = Update + Insert
- **로직**: 유기번호(`desertionNo`)로 DB 조회
    - **없으면?** → 신규 등록 (`INSERT`)
    - **있으면?** → 정보 갱신 (`UPDATE`)
    - **상태가 '종료'면?** → 삭제 (`DELETE`)

### 시각 자료 제안
- 순서도(Flowchart):
  `공공데이터 1건` → `유기번호 조회` → `(Yes/No)` 분기
  → Yes: `dbAnimal.update()`
  → No: `repository.save()`

### 대본 (스크립트)
자, 이제 Part 1의 하이라이트, `AnimalSyncService`입니다. 여기서 **Upsert(업설트)** 라는 중요한 개념이 나옵니다.
업설트는 **Up**date와 In**sert**를 합친 말이에요.
택배 물류센터를 상상해보세요. 물건이 들어왔는데,
1. 처음 보는 물건이면? 새로 진열합니다. (Insert)
2. 이미 있는 물건인데 포장이 바뀌었으면? 새 걸로 바꿔치기합니다. (Update)
3. 만약 "판매 종료"된 상품이면? 진열대에서 치워버립니다. (Delete)
우리 코드는 이 판단을 유기번호(주민번호 같은 것) 하나로 아주 빠르게 수행합니다.

## [Slide 9] 동기화의 핵심 — AnimalSyncService (2)

### 제목
**데이터 가공: 원석을 보석으로**

### 핵심 내용
- **나이 파싱**: "2023(년생)" → `2023` (Integer)
- **체중 파싱**: "5.5(Kg)" → `5.5` (BigDecimal) → **크기 추정** (소형/중형/대형)
- **품종 분류**: "[개] 믹스견" → `DOG`, `믹스견`
- **상태 매핑**: "보호중" → `PROTECTED`, "종료(입양)" → 삭제 대상

### 시각 자료 제안
- Before & After 비교:
  지저분한 원본 데이터("2019(년생)", "4.5Kg") → 깔끔한 정제 데이터(`2019`, `4.5`, `SMALL`)

### 대본 (스크립트)
공공 데이터는 생각보다 많이 지저분합니다. 나이가 "2023(년생)"이라고 괄호까지 섞여서 오고, 체중도 "3.5Kg" 처럼 단위가 붙어있죠.
이걸 그대로 쓰면 통계를 낼 수도, 검색하기도 힘들겠죠?
그래서 우리는 데이터를 저장하기 전에 **'정제(Cleansing)'** 과정을 거칩니다. 숫자만 쏙쏙 뽑아내고, 체중을 보고 "아, 이 정도면 소형견이구나" 하고 크기도 자동으로 분류해줍니다.
이렇게 데이터를 예쁘게 다듬어야 나중에 사용자가 "소형견만 보여줘!" 했을 때 정확하게 보여줄 수 개 있겠죠?

## [Slide 10] DB에 저장 — Animal Entity

### 제목
**우리의 창고: Animal Entity**

### 핵심 내용
- **Entity**: DB 테이블과 1:1로 연결되는 자바 클래스
- **주요 필드**: `species`(개/고양이), `status`(상태), `shelter`(보호소)
- **Shelter 관계**: `ManyToOne` (하나의 보호소 - 여러 마리 동물)

### 시각 자료 제안
- ER 다이어그램 (간략): `Animal` 테이블과 `Shelter` 테이블이 선으로 연결된 그림 (N:1 관계 표시)

### 대본 (스크립트)
이렇게 예쁘게 다듬어진 데이터는 `Animal`이라는 이름표를 달고 데이터베이스 서랍장에 들어갑니다.
여기서 중요한 건 `Shelter`(보호소)와의 관계입니다. 한 보호소에는 여러 마리의 동물이 있겠죠? 그래서 '다대일(N:1)' 관계라고 부릅니다.
동물 정보 안에는 보호소 정보 전체를 다 넣는 게 아니라, "이 동물은 1번 보호소에 있어요"라고 링크(참조)만 걸어둡니다. 이게 바로 관계형 데이터베이스(RDBMS)를 쓰는 효율적인 방법이죠.

## [Slide 11] 스케줄러 자동 동기화

### 제목
**잠든 사이에도 일하는 서버: 스케줄러**

### 핵심 내용
- **Scheduler**: 정해진 시간에 코드 자동 실행
- **설정**: `@Scheduled(cron = "0 0 2 * * *")` → 매일 새벽 2시
- **역할**: 매일 밤, 새로운 유기동물 정보를 업데이트해둠

### 시각 자료 제안
- 시계 아이콘이 새벽 2시를 가리키고, 서버가 혼자 열심히 일하는 귀여운 일러스트
- 코드 하이라이트: `@Scheduled` 어노테이션 부분

### 대본 (스크립트)
우리가 매번 "동기화 버튼"을 누를 순 없겠죠? 우리에겐 **스케줄러**가 있습니다.
매일 새벽 2시, 모두가 잠든 시간. 서버는 조용히 깨어나 공공데이터포털에 다녀옵니다.
"오늘 새로 들어온 친구 있나요?" 하고 물어보고 데이터를 갱신해두죠.

## [Slide 19] 상태 머신 (State Machine)

### 제목
**상태의 흐름: 상태 머신**

### 핵심 내용
- **Enum AdoptionStatus**:
    - `PENDING` (접수 대기)
    - `APPROVED` (승인 완료)
    - `REJECTED` (거절됨)
    - `CANCELLED` (사용자 취소)
- **Rule**: `PENDING` 상태에서만 다른 상태로 변경 가능

### 시각 자료 제안
- 상태 전이 다이어그램 (State Transition Diagram):
  `PENDING`에서 출발하여 3갈래(`APPROVED`, `REJECTED`, `CANCELLED`)로 뻗어나가는 화살표
- 승인된 상태에서 취소로 가는 화살표에 X 표시 (불가)

### 대본 (스크립트)
쇼핑몰에서 주문 배송 조회 해보셨죠?
'결제 완료' → '배송 중' → '배송 완료'. 이게 바로 상태(Status) 관리입니다.
우리 입양 시스템도 똑같습니다. 처음 신청하면 무조건 `PENDING`(대기) 상태입니다.
여기서 관리자가 승인하면 `APPROVED`, 거절하면 `REJECTED`, 사용자가 마음이 바뀌면 `CANCELLED`가 되죠.
중요한 건, 이미 '승인'된 건을 '취소'하거나 다시 '거절'할 수 없다는 겁니다. 한 번 결정되면 끝! 이런 규칙을 코드로 강제하는 게 개발자의 역할입니다.

## [Slide 25] 입양/임시보호 가이드 페이지

### 제목
**친절한 안내서: 가이드 페이지**

### 핵심 내용
- **UX**: 신청 전 절차를 명확히 안내하여 심리적 장벽 낮춤
- **입양 가이드**: 6단계 (녹색 테마)
- **임보 가이드**: 4단계 (파란색 테마)
- **재사용성**: 비슷한 구조지만 색상과 내용만 다르게 구현

### 시각 자료 제안
- 입양 가이드(초록)와 임보 가이드(파랑) 화면 나란히 비교
- 단계별 아이콘 프로세스 바

### 대본 (스크립트)
신청 버튼 누르기 전에 "입양은 어떻게 진행되나요?" 궁금해하는 분들을 위해 가이드 페이지를 따로 만들었습니다.
입양은 녹색, 임시보호는 파란색으로 테마를 다르게 해서 헷갈리지 않게 했고요.
단계별로 아이콘을 써서 1단계 교육, 2단계 상담... 이렇게 게임 퀘스트 깨듯이 보여줍니다.
사용자가 막막해하지 않고 편안하게 신청할 수 있도록 돕는 장치입니다.

## [Slide 26] 전화 연결 컴포넌트 (GuideCallBlock)

### 제목
**작은 컴포넌트의 큰 역할: GuideCallBlock**

### 핵심 내용
- **기능**: 보호소 정보 + 유기번호 표시 + 복사 기능
- **재사용**: 입양/임보 가이드 양쪽에서 똑같이 사용
- **UX 디테일**: 클릭 한번으로 유기번호 복사 (전화 문의 시 필수)

### 시각 자료 제안
- 전화번호 박스 컴포넌트 확대
- "복사되었습니다!" 토스트 메시지

### 대본 (스크립트)
여기 작지만 기특한 녀석이 있습니다. 바로 '전화 연결 박스'인데요.
보호소에 전화 걸 때 제일 먼저 물어보는 게 "유기번호가 뭐예요?"거든요. 긴 번호 외우기 힘들잖아요?
그래서 옆에 '복사' 버튼을 달아줬습니다.
입양 페이지에도, 임보 페이지에도 똑같이 들어가는 이 박스, 컴포넌트 하나 잘 만들어두면 여기저기서 재사용하니 개발 효율도 쑥쑥 오릅니다.

## [Slide 27] 전체 시퀀스 다이어그램

### 제목
**한눈에 보는 전체 흐름 (Summary)**

### 핵심 내용
1. `User`: 상세 페이지 진입
2. `Frontend`: 입양 가이드 확인 → 신청 폼 작성 및 제출
3. `Backend`: 신청서 저장(PENDING) → 이메일 발송
4. `Admin`: 신청서 확인 → 승인(APPROVED)
5. `Backend`: 승인 처리 → DB 갱신 → 결과 통보
6. `User`: 입양 확정 및 동물 인수

### 시각 자료 제안
- 전체 과정을 아우르는 시퀀스 다이어그램 (Swimlane: User | Frontend | Backend | Database | Admin)

### 대본 (스크립트)
자, 이제 마무리를 향해 갑니다. 우리가 만든 전체 시스템이 어떻게 돌아가는지 영상 보듯이 머릿속으로 그려보세요.
사용자가 동물을 보고 반해서 신청서를 쓰고, 서버가 접수하고, 관리자가 승인 도장을 쾅! 찍으면, 다시 사용자에게 기쁜 소식이 날아가는 이 과정.
이 모든 게 물 흐르듯이 이어집니다. Part 1의 데이터 동기화가 없었다면 동물도 없었을 테고, Part 2의 입양 시스템이 없었다면 만남은 성사되지 않았겠죠.

## [Slide 28] 검증 & 에러 핸들링

### 제목
**완벽을 위한 마무리: 검증과 에러 처리**

### 핵심 내용
- **Validation**: `@Valid`로 필수 입력값 체크 (빈칸 방지)
- **Exception**: "이미 신청한 동물입니다", "존재하지 않는 동물입니다"
- **API Key Check**: 서버 시작 시 키 설정 여부 확인

### 시각 자료 제안
- 에러 메시지 팝업 예시 ("사유를 입력해주세요!")
- 자바 코드: `@NotNull`, `orElseThrow()`

### 대본 (스크립트)
하지만 현실 세계엔 예외가 많습니다. 사유를 안 적고 신청 버튼을 누르거나, 이미 누가 입양해 간 동물을 또 신청할 수도 있죠.
그래서 우리는 꼼꼼한 방어막을 쳐뒀습니다.
"빈칸은 안 돼요!", "이 동물은 이미 입양 갔어요!"라고, 서버가 깐깐하게 체크해서 알려줍니다.
이런 에러 처리가 잘 되어 있어야 사용자가 당황하지 않고 올바르게 서비스를 이용할 수 있습니다.

## [Slide 29] 실무 설계 포인트 총정리

### 제목
**오늘 배운 실무 포인트**

### 핵심 내용
1. **외부 의존성 격리**: API 응답 변경에 대비한 DTO/Wrapper 분리
2. **동기화 전략**: Upsert로 데이터 정합성 유지
3. **통합 모델링**: 유사 기능(입양/임보)은 하나의 엔티티 + Enum으로 관리
4. **상태 관리**: 명확한 상태 전이 규칙(State Machine) 적용

### 시각 자료 제안
- 4가지 포인트를 아이콘과 함께 카드 형태로 요약

### 대본 (스크립트)
오늘 강의를 요약해볼까요?
첫째, 남의 집(외부 API) 데이터는 언제 바뀔지 모르니 우리 식(DTO)으로 잘 포장해서 쓰자.
둘째, 데이터 동기화는 '없으면 넣고, 있으면 고치고, 사라지면 지우는' Upsert가 답이다.
셋째, 비슷한 기능은 따로 만들지 말고 하나로 합치되, Enum으로 구분하자.
넷째, 상태(Status)는 함부로 바꾸지 못하게 규칙을 정하자.
이 네 가지만 기억해도 여러분은 이미 주니어 개발자 이상의 설계를 할 줄 아는 겁니다.
