1️⃣ Spring Security 단독 vs JWT 토큰

Spring Security 단독 사용 (세션 기반)

로그인 → 서버에 세션 생성 → JSESSIONID 쿠키 발급 → 매 요청마다 쿠키로 세션 확인
특징:
  • 서버 메모리에 세션 저장 (Stateful)
  • 쿠키에 세션 ID만 저장
  • 서버가 세션을 직접 관리 (로그아웃 시 즉시 무효화 가능)
  • 서버 증설 시 세션 공유 필요 (Redis 등)

JWT 토큰 방식

로그인 → JWT 발급 → 클라이언트가 토큰 저장 → 매 요청마다 토큰 전송
특징:
  • 서버에 상태 저장 안 함 (Stateless)
  • 토큰 자체에 사용자 정보 포함
  • 서버는 서명만 검증
  • 서버 증설 자유로움

비교표

구분 세션 (Spring Security 단독) JWT
저장 위치 서버 메모리/Redis 클라이언트 (LocalStorage/Cookie)
상태 관리 Stateful Stateless
확장성 서버 증설 시 세션 공유 필요 서버 증설 자유로움
로그아웃 즉시 무효화 가능 만료까지 유효 (블랙리스트 필요)
보안 취약점 CSRF 공격 XSS 공격
서버 부하 매 요청마다 세션 조회 서명 검증만 (DB 조회 없음)
모바일 앱 쿠키 관리 어려움 토큰 관리 쉬움

어떤 게 더 나은가?

✅ JWT가 유리한 경우 (현재 프로젝트):
  • MSA (마이크로서비스) 아키텍처
  • 모바일 앱 + 웹 동시 지원
  • 서버 확장이 자주 필요한 경우
  • API 서버 (프론트엔드 분리)
  • REST API 중심
세션이 유리한 경우:
  • 즉시 로그아웃 필요 (보안 매우 중요)
  • 단일 서버 환경
  • 전통적인 웹 애플리케이션 (SSR)
⭐ 현재 트렌드: JWT가 대세! 특히 REST API, SPA, 모바일 앱에서는 거의 표준입니다.

2️⃣ HMAC-SHA 알고리즘과 JWT_SECRET

HMAC-SHA란?

HMAC = Hash-based Message Authentication Code

SHA = Secure Hash Algorithm

JWT 서명 = HMAC-SHA256(Header + Payload, Secret Key)

동작 원리

1. JWT 생성

// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload
{
  "sub": "user@example.com",
  "role": "USER",
  "userId": 123,
  "exp": 1735689600
}

// Signature 생성
signature = HMAC_SHA256(
  base64(header) + "." + base64(payload),
  "my-secret-key-12345"
)

// 최종 JWT
eyJhbGc... . eyJzdWI... . a8f3d9e...

2. JWT 검증

// 받은 JWT를 . 으로 분리
[header, payload, signature] = jwt.split('.')

// 같은 방식으로 서명 재생성
newSignature = HMAC_SHA256(
  header + "." + payload,
  "my-secret-key-12345"
)

// 비교
if (newSignature === signature) {
  // 검증 성공 (위변조 없음)
} else {
  // 검증 실패 (위변조됨)
}

JWT_SECRET vs OAuth Secret Key

⚠️ 완전히 다릅니다!

JWT_SECRET (현재 프로젝트)

# application.yml
jwt:
  secret: your-256-bit-secret-key  # 내가 만든 비밀 키
  • 용도 JWT 서명/검증에 사용
  • 관리 내 서버에서만 사용
  • 생성 내가 직접 랜덤 문자열 생성
  • 생성 방법 openssl rand -base64 32

OAuth Secret Key (구글/카카오)

# application.yml
KAKAO_CLIENT_SECRET: abc123...  # 카카오가 발급한 키
GOOGLE_CLIENT_SECRET: xyz789... # 구글이 발급한 키
  • 용도 OAuth 인증 시 카카오/구글 서버와 통신
  • 관리 카카오/구글 개발자 콘솔에서 발급
  • 생성 카카오/구글이 발급

전체 흐름 비교

일반 로그인

1. 사용자: 이메일/비밀번호 입력 2. 서버: 비밀번호 검증 3. 서버: JWT 발급 (JWT_SECRET 사용) ← 여기! 4. 클라이언트: JWT 저장

OAuth 로그인 (카카오)

1. 사용자: 카카오 로그인 클릭 2. 카카오: 로그인 페이지 표시 3. 사용자: 카카오 로그인 4. 카카오: Authorization Code 발급 5. 내 서버: Code로 Access Token 요청 (KAKAO_CLIENT_SECRET 사용) 6. 카카오: Access Token 발급 7. 내 서버: Access Token으로 사용자 정보 요청 8. 카카오: 사용자 정보 반환 9. 내 서버: JWT 발급 (JWT_SECRET 사용) ← 여기! 10. 클라이언트: JWT 저장
핵심:
  • OAuth Secret은 카카오/구글과 통신할 때만 사용
  • JWT_SECRET은 내 서버의 JWT를 만들 때만 사용
  • 둘은 완전히 별개!

3️⃣ JWT Filter 검증 시점

⚠️ 중요: 모든 HTTP 요청마다 검증합니다!
화면 이동이 아니라 API 호출마다!

예시 시나리오

사용자 행동 API 호출 JWT Filter 실행
로그인 POST /api/auth/login ❌ (Public Path)
메인 페이지 접속 (화면 이동) 없음 (React Router) ❌
동물 목록 조회 GET /api/animals ✅ (검증하지만 Public)
동물 상세 클릭 (화면 이동) 없음 ❌
동물 상세 정보 조회 GET /api/animals/123 ✅ (검증하지만 Public)
입양 신청 버튼 클릭 POST /api/adoptions ✅ (검증 필수, Protected)
내 정보 페이지 이동 (화면 이동) 없음 ❌
내 정보 조회 GET /api/users/me ✅ (검증 필수, Protected)
핵심:
  • 화면 이동 ≠ API 호출
  • JWT Filter는 API 호출할 때만 실행
  • 프론트엔드 라우팅은 백엔드와 무관

JWT Filter 동작 과정

// JwtAuthenticationFilter.java
protected void doFilterInternal(
    HttpServletRequest request,
    HttpServletResponse response,
    FilterChain filterChain) {
    
    try {
        // 1. Authorization 헤더에서 토큰 추출
        String token = resolveToken(request);
        // "Authorization: Bearer eyJhbGc..." → "eyJhbGc..."
        
        // 2. 토큰이 있고 유효한지 확인
        if (StringUtils.hasText(token) && 
            jwtProvider.validateAccessToken(token)) {
            
            // 3. 토큰에서 정보 추출
            Claims claims = jwtProvider.parseAccessToken(token);
            String email = claims.getSubject();
            String role = claims.get("role", String.class);
            Long userId = claims.get("userId", Long.class);
            
            // 4. Spring Security 인증 객체 생성
            Authentication authentication = 
                new UsernamePasswordAuthenticationToken(
                    email, null, 
                    Collections.singletonList(
                        new SimpleGrantedAuthority("ROLE_" + role)
                    )
                );
            
            // 5. SecurityContext에 저장
            SecurityContextHolder.getContext()
                .setAuthentication(authentication);
            
            // 6. request에 userId 저장
            request.setAttribute("userId", userId);
        }
    } catch (Exception ignored) {
        // 토큰 무효 → 인증 없이 진행
    }
    
    // 7. 다음 필터로 진행
    filterChain.doFilter(request, response);
}

Public Path vs Protected Path

// SecurityConfig.java
private static final String[] PUBLIC_PATHS = {
    "/api/auth/signup",      // 회원가입
    "/api/auth/login",       // 로그인
    "/api/animals",          // 동물 목록 (공개)
    "/api/boards",           // 게시판 (공개)
};

@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) {
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers(PUBLIC_PATHS).permitAll()  // 인증 불필요
        .requestMatchers("/api/**").authenticated() // 인증 필요
    );
}
  • Public Path: JWT 없어도 접근 가능 (하지만 Filter는 실행됨)
  • Protected Path: JWT 필수, 없으면 401 Error

4️⃣ 일반 회원가입과 Refresh Token

모든 로그인 방식의 최종 목표: JWT 발급

방식 과정 JWT 발급
일반 회원가입 User 생성 → 회원가입 완료 ❌ (이후 로그인 필요)
일반 로그인 비밀번호 검증 → JWT 발급 ✅
구글 로그인 구글 토큰 검증 → User 조회/생성 → JWT 발급 ✅
카카오 로그인 카카오 토큰 검증 → User 조회/생성 → JWT 발급 ✅

회원가입 vs 로그인

일반 회원가입 (JWT 발급 안 함)

// AuthService.java
public UserResponse signup(SignupRequest request) {
    // 1. 이메일 중복 체크
    if (userRepository.existsByEmail(email)) {
        throw new CustomException("이미 사용 중인 이메일");
    }
    
    // 2. 비밀번호 해싱
    String encodedPassword = passwordEncoder.encode(password);
    
    // 3. User 생성
    User user = User.builder()
        .email(email)
        .password(encodedPassword)
        .name(name)
        .role(Role.USER)
        .build();
    
    // 4. DB 저장
    userRepository.save(user);
    
    // 5. UserResponse 반환 (JWT 발급 안 함!)
    return toUserResponse(user);
}

일반 로그인 (JWT 발급)

// AuthService.java
public TokenResponse login(LoginRequest request) {
    // 1. 사용자 조회
    User user = userRepository.findByEmail(email)
        .orElseThrow(() -> new UnauthorizedException("이메일 또는 비밀번호 오류"));
    
    // 2. 비밀번호 검증
    if (!passwordEncoder.matches(password, user.getPassword())) {
        throw new UnauthorizedException("이메일 또는 비밀번호 오류");
    }
    
    // 3. JWT 발급
    String accessToken = jwtProvider.createAccessToken(
        user.getEmail(), 
        user.getRole().name(), 
        user.getId()
    );
    String refreshToken = jwtProvider.createRefreshToken(user.getEmail());
    
    // 4. TokenResponse 반환
    return TokenResponse.builder()
        .accessToken(accessToken)
        .refreshToken(refreshToken)
        .tokenType("Bearer")
        .expiresIn(86400) // 24시간
        .user(toUserResponse(user))
        .build();
}
차이점:
  • 회원가입: User만 생성, JWT 발급 안 함
  • 로그인: JWT 발급 (일반/구글/카카오 모두 동일)

Refresh Token의 역할

문제 상황:
Access Token 만료 시간: 24시간
→ 24시간마다 다시 로그인? 불편함!
해결책:
Refresh Token 만료 시간: 30일
→ Access Token 만료되면 Refresh Token으로 재발급

Refresh Token 동작 코드

// AuthService.java
public TokenResponse refreshToken(String refreshToken) {
    // 1. Refresh Token 검증
    if (!jwtProvider.validateRefreshToken(refreshToken)) {
        throw new UnauthorizedException("유효하지 않은 갱신 토큰");
    }
    
    // 2. 토큰에서 이메일 추출
    Claims claims = jwtProvider.parseRefreshToken(refreshToken);
    String email = claims.getSubject();
    
    // 3. 사용자 조회
    User user = userRepository.findByEmail(email)
        .orElseThrow(() -> new UnauthorizedException("사용자 없음"));
    
    // 4. 새로운 토큰 발급
    String newAccessToken = jwtProvider.createAccessToken(
        user.getEmail(), 
        user.getRole().name(), 
        user.getId()
    );
    String newRefreshToken = jwtProvider.createRefreshToken(user.getEmail());
    
    // 5. 반환
    return TokenResponse.builder()
        .accessToken(newAccessToken)
        .refreshToken(newRefreshToken)
        .build();
}

실제 사용 흐름

Day 1 로그인 → Access Token (24h) + Refresh Token (30d) Day 2 - Access Token 만료 API 호출 → 401 Error → 프론트엔드가 자동으로 Refresh Token으로 재발급 요청 → POST /api/auth/refresh → 새 Access Token + Refresh Token 받음 → 원래 API 재시도 Day 31 - Refresh Token도 만료 → 다시 로그인 필요

Refresh Token 보안

⚠️ 현재 프로젝트의 한계:
  • Refresh Token을 DB에 저장 안 함!
  • → 탈취되면 만료까지 막을 방법 없음
✅ 개선 방안: 
// Redis에 Refresh Token 저장
redis.set("refresh:user123", refreshToken, 30일);

// 로그아웃 시
redis.delete("refresh:user123");  // 즉시 무효화!

// 재발급 시
if (!redis.exists("refresh:user123")) {
    throw new UnauthorizedException("로그아웃된 토큰");
}

5️⃣ 스터디 추천 주제 상세

1. JWT 심화

JWT 구조

// Header (알고리즘 정보)
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload (실제 데이터)
{
  "sub": "user@example.com",
  "role": "USER",
  "userId": 123,
  "exp": 1735689600
}

// Signature (서명)
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)

// 최종 JWT
eyJhbGc...  (Header)
.
eyJzdWI...  (Payload)
.
a8f3d9e...  (Signature)

HS256 vs RS256

구분 HS256 (현재 프로젝트) RS256
알고리즘 HMAC + SHA256 (대칭키) RSA + SHA256 (비대칭키)
키 Secret Key 1개 Private Key + Public Key
서명 Secret Key로 서명 Private Key로 서명
검증 Secret Key로 검증 Public Key로 검증
속도 빠름 느림
보안 Secret Key 유출 시 위험 Private Key만 안전하게 보관
사용처 단일 서버 MSA (여러 서버)

2. 암호화

BCrypt vs Argon2 vs PBKDF2

알고리즘 특징 속도 보안 사용처
BCrypt Salt 자동 생성 느림 높음 현재 프로젝트
Argon2 메모리 하드 매우 느림 최고 최신 프로젝트
PBKDF2 표준 알고리즘 중간 중간 레거시

Salt와 Pepper

Salt: DB에 저장 (사용자마다 다름) 
password123 + salt1 → hash → abc123def456
password123 + salt2 → hash → xyz789ghi012 (다름!)
Pepper: 환경변수에 저장 (모든 사용자 공통) 
최종 해시 = BCrypt(password + pepper, salt)

Rainbow Table 공격

Rainbow Table: 미리 계산된 해시 테이블 
password123 → abc123def456
qwerty → xyz789ghi012
...수백만 개
방어: Salt 사용 → Rainbow Table 무용지물
(같은 비밀번호도 다른 해시)

3. OAuth 2.0

Authorization Code Flow (현재 프로젝트)

1. 사용자: 카카오 로그인 클릭 2. 클라이언트: 카카오 인가 서버로 리다이렉트 3. 카카오: 로그인 페이지 표시 4. 사용자: 카카오 로그인 5. 카카오: Authorization Code 발급 6. 클라이언트: Code를 백엔드로 전달 7. 백엔드: Code로 Access Token 요청 (Client Secret 사용) 8. 카카오: Access Token 발급 9. 백엔드: Access Token으로 사용자 정보 요청 10. 카카오: 사용자 정보 반환 11. 백엔드: JWT 발급 12. 클라이언트: JWT 저장

Access Token vs ID Token

구분 Access Token ID Token
용도 API 호출 권한 사용자 인증
포함 정보 권한 범위 사용자 정보
형식 불투명 문자열 JWT
검증 Resource Server Client
예시 카카오 API 호출 구글 로그인
현재 프로젝트:
  • 구글: ID Token 사용 (사용자 정보 포함)
  • 카카오: Access Token으로 사용자 정보 조회

4. Spring Security

Filter Chain 동작 원리

Request ↓ SecurityContextPersistenceFilter (SecurityContext 로드/저장) ↓ LogoutFilter (로그아웃 처리) ↓ UsernamePasswordAuthenticationFilter (폼 로그인) ↓ JwtAuthenticationFilter (JWT 검증 - 커스텀) ↓ ExceptionTranslationFilter (인증/인가 예외 처리) ↓ FilterSecurityInterceptor (최종 권한 확인) ↓ Controller

SecurityContext와 ThreadLocal

// SecurityContext: 현재 인증 정보 저장소
SecurityContext context = SecurityContextHolder.getContext();
Authentication auth = context.getAuthentication();

// ThreadLocal: 스레드마다 독립적인 저장소
// → 요청마다 다른 스레드 → 다른 SecurityContext

Request 1 (Thread 1) → SecurityContext 1 (user1)
Request 2 (Thread 2) → SecurityContext 2 (user2)
→ 서로 간섭 없음!

@PreAuthorize vs @Secured

// @PreAuthorize (SpEL 사용 가능)
@PreAuthorize("hasRole('ADMIN')")
@PreAuthorize("hasRole('ADMIN') or hasRole('USER')")
@PreAuthorize("#userId == authentication.principal.userId")

// @Secured (단순)
@Secured("ROLE_ADMIN")
@Secured({"ROLE_ADMIN", "ROLE_USER"})

5. 보안 패턴

CSRF vs XSS

CSRF (Cross-Site Request Forgery): 
공격자 사이트 → 내 사이트로 요청 위조
예: <img src="https://bank.com/transfer?to=hacker&amount=1000">
방어:
  • CSRF Token 사용
  • 쿠키 + 폼에 토큰 포함
  • 둘이 일치해야 요청 처리
REST API는 왜 CSRF 방어 불필요?
  • 쿠키 사용 안 함 (JWT는 Header로 전송)
  • Same-Origin Policy로 차단
  • CORS로 허용된 출처만 접근
XSS (Cross-Site Scripting): 
악성 스크립트 삽입
예: <script>alert(document.cookie)</script>
방어:
  • 입력값 검증 (Validation)
  • 출력 시 이스케이프 (<script> → &lt;script&gt;)
  • Content Security Policy (CSP)
  • HttpOnly 쿠키 (JS 접근 차단)

CORS 동작 원리

Preflight Request (사전 요청): Browser → API: OPTIONS /api/users API → Browser: Access-Control-Allow-Origin: localhost:5173 실제 요청: Browser → API: GET /api/users API → Browser: 200 OK + 데이터
Preflight Request: 브라우저가 자동으로 보내는 사전 요청
→ 서버가 허용하는지 확인
→ 허용하면 실제 요청 전송

Refresh Token Rotation (RTR)

기존 방식 (현재 프로젝트):
  • Refresh Token 재사용 가능
  • → 탈취되면 만료까지 계속 사용 가능
RTR 방식:
  • Refresh Token 사용 시마다 새로 발급
  • → 이전 Refresh Token 무효화
  • → 탈취된 토큰 사용 시 감지 가능
// RTR 구현
public TokenResponse refreshToken(String oldRefreshToken) {
    // 1. 검증
    validateRefreshToken(oldRefreshToken);
    
    // 2. 새 토큰 발급
    String newAccessToken = createAccessToken(...);
    String newRefreshToken = createRefreshToken(...);
    
    // 3. 이전 Refresh Token 블랙리스트 등록
    redis.set("blacklist:" + oldRefreshToken, "used", 30일);
    
    // 4. 새 Refresh Token 저장
    redis.set("refresh:user123", newRefreshToken, 30일);
    
    return new TokenResponse(newAccessToken, newRefreshToken);
}

📚 핵심 요약

  • JWT vs 세션: JWT는 Stateless, 확장성 좋음, 현재 트렌드
  • HMAC-SHA: JWT 서명 알고리즘, Secret Key로 위변조 방지
  • JWT_SECRET: 내 서버의 JWT 생성용, OAuth Secret과 별개
  • JWT Filter: 모든 API 호출마다 검증 (화면 이동 아님)
  • 회원가입: JWT 발급 안 함, 로그인 시에만 발급
  • Refresh Token: Access Token 재발급용, 30일 유효
  • 보안: BCrypt 해싱, CSRF/XSS 방어, CORS 설정

⚠️ 주의사항

  • JWT_SECRET은 절대 코드에 하드코딩 금지
  • Refresh Token은 DB/Redis에 저장 권장
  • 로그아웃 시 블랙리스트 구현 필요
  • XSS 방어를 위해 입력값 검증 필수
